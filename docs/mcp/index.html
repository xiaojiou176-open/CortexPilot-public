Read-only MCP quickstart

CortexPilot already exposes a real Model Context Protocol (MCP) surface, but the current contract is intentionally read-only. This page exists so people searching for MCP, Codex, and Claude Code integrations can find the truthful surface quickly instead of guessing from source files.

What the current MCP surface is for

  • inspect runs, Workflow Cases, approvals, compare outputs, proof packs, and incident summaries
  • return structured JSON that other tools and agents can consume safely
  • keep operator truth readable without creating a second control plane

Fast adoption path

  1. Use this page when you want the shortest truthful explanation of what the current MCP surface exposes.
  2. Jump to the compatibility matrix if your team first needs the “which adoption ladder fits our stack?” answer.
  3. Jump to the API and contract quickstart when you also need OpenAPI, route/type boundaries, or the thin client package.

  5. Jump to the integration guide or the skills quickstart when your real question is how a coding-agent team should adopt this surface truthfully.

Try the current server

The current server is a repo-owned stdio JSON-RPC node. Run it from the repo root when you want the shortest proof that the public MCP story is real: 

PYTHONPATH=apps/orchestrator/src python3 -m cortexpilot_orch.cli mcp-readonly-server

Current tool set: list_runs, get_run, get_run_events, get_run_reports, list_workflows, get_workflow, list_queue, get_pending_approvals, get_diff_gate_state, get_compare_summary, get_proof_summary, and get_incident_summary.

What it does not do yet

  • no write-capable MCP mutations
  • no hosted operator account or managed login layer
  • no claim that MCP replaces the orchestration runtime itself

Why this matters for Codex and Claude Code workflows

MCP is the connective tissue that lets external tools inspect workflow truth without granting them control they do not yet have. That is useful for teams running Codex-style or Claude Code-style operator loops, because you can keep evidence readable without pretending the system is already a write-enabled MCP platform.

Need the surrounding HTTP/package map as well? Pair this page with the API and contract quickstart and the builder quickstart so protocol readers and frontend builders stay on the same truthful boundary.

FAQ

  • Is this a real MCP server? Yes. The current public contract is read-only, but the server surface itself is real and runnable from the repo.
  • Does MCP support writes? No. The public promise remains read-only even though guarded queue/operator mutation groundwork exists elsewhere in the repo.
  • Should teams start here or with API/package docs? Start here when protocol inspection is the first job; move to API or builder docs when package or HTTP integration becomes the real next step.
