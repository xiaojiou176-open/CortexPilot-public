Builder quickstart

CortexPilot is not a full SDK platform today, but it does already expose a useful builder surface across three layers.

Choose the shortest truthful entrypoint

  • Start with the read-only MCP quickstart when you want protocol-level inspection and frozen write-boundary context.
  • Start with the API and contract quickstart when you want OpenAPI, route/type boundaries, and the thin client story.
  • Come back here when you are ready to choose which package layer to import into dashboard, desktop, or another frontend.

What builders can do today

  • Call control-plane reads from the thin frontend API client.
  • Reuse generated contract-facing types without importing backend modules.
  • Share locale, status-presentation, and UI copy substrate across dashboard and desktop consumers.
  • Compose those surfaces with the existing read-only MCP story instead of inventing a second integration contract.

1. Frontend API client

Start with @cortexpilot/frontend-api-client when you want ready-to-call helpers for runs, Workflow Cases, approvals, and command-tower reads.

The current shortest integration path is the package's createControlPlaneStarter(...) helper, which bootstraps Command Tower overview + agents + contracts + optional role config reads without inventing a second client wrapper or implying hosted operator accounts.

Current boundaries

  • This is a builder entrypoint layer, not a hosted operator product.
  • The MCP story is real, but the current public boundary stays read-only.
  • The package surface is useful today, but it should not be described as a complete SDK platform.
  • The current packages remain repo-owned/private integration surfaces; do not describe them as published marketplace artifacts or official plugin packages.
  • Any repo-owned mutation helpers below are operator-only control-plane add-ons, not the default external builder onboarding contract.

Minimal example

import {
  createControlPlaneStarter,
  createFrontendApiClient,
} from "@cortexpilot/frontend-api-client";

const client = createFrontendApiClient({
  baseUrl: "http://localhost:8000",
  resolveMutationRole: () => "TECH_LEAD",
});

const starter = createControlPlaneStarter(client);
const bootstrap = await starter.fetchBootstrap({ role: "WORKER" });
const preview = await starter.previewRoleDefaults("WORKER", {
  runtime_binding: { provider: "cliproxyapi", model: "gpt-5.4" },
});

Repo-owned starter file

If you want a path you can run inside this repo without inventing your own wrapper first, use the control-plane starter example. 

node packages/frontend-api-client/examples/control_plane_starter.local.mjs \
  --base-url http://127.0.0.1:10000 \
  --role WORKER \
  --mutation-role TECH_LEAD \
  --preview-provider cliproxyapi \
  --preview-model gpt-5.4

This remains a repo-owned starter path, not a hosted SDK. It proves one outside-consumer loop for overview + agents + contracts + role-config preview without claiming hosted accounts, write-capable MCP, or a full marketplace/distribution product.

Guarded operator-only add-ons

Some repo-owned helpers sit next to the builder path, but they should be read as trusted maintainer tools rather than default external onboarding.

The optional --apply flag is an operator-only, repo-owned control-plane action for trusted maintainers working inside this repository. It is outside the default external builder contract and does not mean write-capable MCP or general external mutation is available.

The base client also carries queue preview/cancel helpers for the same trusted repo-operator policy layer. Keep that story separate from the default “read the protocol, then adopt the packages” builder ladder.

Apply stays opt-in. Pass --apply only when you intentionally want a trusted operator flow to persist repo-owned role defaults inside this repo.
