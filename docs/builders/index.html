Builder quickstart

CortexPilot is not a full SDK platform today, but it does already expose a useful builder surface across three layers.

Choose the shortest truthful entrypoint

  • Start with the read-only MCP quickstart when you want protocol-level inspection and frozen write-boundary context.
  • Start with the API and contract quickstart when you want OpenAPI, route/type boundaries, and the thin client story.
  • Come back here when you are ready to choose which package layer to import into dashboard, desktop, or another frontend.

What builders can do today

  • Call control-plane reads from the thin frontend API client.
  • Reuse generated contract-facing types without importing backend modules.
  • Share locale, status-presentation, and UI copy substrate across dashboard and desktop consumers.
  • Compose those surfaces with the existing read-only MCP story instead of inventing a second integration contract.

1. Frontend API client

Start with @cortexpilot/frontend-api-client when you want ready-to-call helpers for runs, Workflow Cases, approvals, and command-tower reads.

Open package README

2. Frontend API contract

Use @cortexpilot/frontend-api-contract when you want the generated contract-facing type boundary and route/query names without importing backend modules.

Open generated contract types

3. Shared frontend substrate

Use @cortexpilot/frontend-shared for brand copy, locale, status presentation, and frontend-only shared types.

Open shared README

Current boundaries

  • This is a builder entrypoint layer, not a hosted operator product.
  • The MCP story is real, but the current public boundary stays read-only.
  • The package surface is useful today, but it should not be described as a complete SDK platform.

Minimal example

import { createFrontendApiClient } from "@cortexpilot/frontend-api-client";

const client = createFrontendApiClient({
  baseUrl: "http://localhost:8000",
});

const overview = await client.fetchCommandTowerOverview();
const workflows = await client.fetchWorkflows();

Fast onboarding order

  1. Read MCP quickstart for the truthful read-only protocol boundary.
  2. Read API and contract quickstart for OpenAPI + type/package entrypoints.
  3. Pick the package layer below only after you know whether you need protocol reads, HTTP reads, or shared UI substrate.
